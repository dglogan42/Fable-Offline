# Goal quality (for offline loops)

**Consumers:** Humans writing goals; Hermes/Fable when restating or refining a goal.

## Good goals

| Property | Example |
|----------|---------|
| **Done definition** | “FastAPI todo API with pytest green and README run steps” |
| **Artifact** | Code, memo, table, checklist — something checkable |
| **Scope bound** | One product surface, not “fix the product and market” |
| **Success criteria** | 3–7 criteria, scorable, not vibes-only |

## Bad goals

| Pattern | Why loops fail |
|---------|----------------|
| “Make it good” | No verifier |
| “Research everything about X” | No stop |
| Multiple unrelated jobs | Context thrash |
| No artifact | Nothing to grade |

## Success condition templates

- **Build:** “Runnable from README; tests pass; no secrets in repo”  
- **Audit:** “Verdict first; evidence table; OPEN unknowns named”  
- **Memo:** “Verdict; three risks; numbers re-derived or labeled”  

## Agent duty

If the human goal is vague, **restate a checkable version** in the first cycle’s UNIT or OPEN — do not silently invent a different goal.
